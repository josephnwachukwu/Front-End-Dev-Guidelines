Front End Dev Guidelines and Best Practices

This document contains what i believe to be the best practices for front-end web development Each item here represents either:

  1. A reminder to follow existing standards or industry conventions,
  2. guidance on what constitutes professional patterns and organization, or
  3. a decision I have made favoring one method over its alternatives.

CSS

Avoid using inline styles

You don't want to have to change code in many places.

Avoid using the important! tag

You should not use the important tag because it interferes with the natural hierarchy of css selectors.

Less vs Sass vs Stylus

All three are CSS preprocessors. Less and sass use regular css syntax while styus omits brackets and colons.

Use BEM!

BEM Block Element Modifier is a methodology that helps speed development individually and in teams by making code much more readable and by making css styles easier to group. Bem can also be applied to other computer problems like file systems. /* Block component */ .btn {} /* Element that depends upon the block */ .btn__cancel {} /* Modifier that changes the style of the block */ .btn--danger {} .btn--big {}

Always use transitions over javascript animations

The reason for using css3 transformations over javascript animations is performance. First of all you offload all the graphics calculations to a different thread. Because css3 uses hardware acceleration you will be using the GPU instead. The performance increase is not much but it is some. Not all css animations benefit from the GPU boost also Transforms (scale, rotation, translation, and skew) and opacity are the main processes that use it.

Javascript

ES5, ES6, Typescript

  • ES5 is the JavaScript you know and use in the browser today it is what it is and does not require a build step to transform it into something that will run in today's browsers
  • ES6 is the next iteration of JavaScript, but it does not run in today's browsers. There are quite a few transpilers that will export ES5 for running in browsers. It is still an dynamic (read: untyped) language.
  • TypeScript provides an optional typing system while pulling in features from future versions of JavaScript (ES6 and ES7).
Note: a lot of the transpilers out there (i.e. babel, TypeScript) will allow you to use features from future versions of JavaScript today and exporting code that will still run in today's browsers.
Loading in vs Put scripts in the when they should happen before the page starts being rendered. Examples are things like HTML5 shim libraries or Modernizr. Put scripts in "ready" handlers when there's limited control over how the script is imported. Things like utilities that can be included on a page to unobtrusively add features generally use "ready" handlers because they can't be sure of how they'll be used. Put scripts at the end of the if you can get away with it in all other cases.

Always use the ‘use strict” directive

The "use strict" directive is a literal expression ignored by earlier versions of JavaScript before 1.8.5 (ECMAScript version 5). The purpose of "use strict" is to indicate that the code should be executed in "strict mode".

Avoid globals

Everyone knows you're not supposed to use the global scope for functions and variables because every js file loaded uses the same scope. Function names and variables can clash and be overwritten with your stuff. Use an object literal instead var header = { Init: function() {}, Login: function() {}, userId:null }; Yes its a pain sometimes to have to use the namespace prefix before you call your function header.init() But it's better coding. Always cache the array length when looping through an array. You don't want to keep calculating array length each time you go through the for loop and sometimes the array is manipulated and the length changes. for (var i = 0, len = arr.length; i < len; i++) { //code here }

Development code is not live code

Code should be minimized and stripped of all comments. Using something like jsLint.

CDN vs Local Concatenated Files

The value of a CDN lies in the likelihood of the user having already visited another site calling that same file from that CDN, and becomes increasingly valuable depending on the size of the file. The likelihood of this being the case increases with the ubiquity of the file being requested and the popularity of the CDN. With this in mind, pulling a relatively large and popular file from a popular CDN makes absolute sense. jQuery, and, to a lesser degree, jQuery UI, fit this bill. Meanwhile, concatenating files makes sense for smaller files which are not likely to change much — your commonly used plugins will fit this bill, but your core application-specific code probably doesn't: it might change from week to week, and if you're concatenating it with all your other files, you'd have to force the user to download everything all over again. I actually split it half and half. I get the popular stuff from the cdn like jQuery, Angular, Modernizr etc. Then i concat all my js and various third party plugins.

Node best practices

Start with npm init

This builds you valid package.json for your project, inferring common properties from the working directory.

Watch your modules

When you are in development you may want your apps node modules to update but when the app is out in production drifting modules can be dangerous. Especially if a develope pulls a module out or changes something drastic.

Simplify your build

Rather than picking between the best task runner or inheriting large complex build try and simplify your build by using a ‘vanilla’ build without bower, gulp, or grunt.
